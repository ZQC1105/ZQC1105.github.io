2. 四种“只读+序列化”出路速查表
------------------------------------------------
| 方案                     | 语法变化 | 编译期必填 | 真只读 | 无参构造 | 原生 Json | 推荐场景                   |
| ------------------------ | -------- | ---------- | ------ | -------- | --------- | -------------------------- |
| ① `required init`        | 无       | ✅          | ✅      | ✅        | ✅         | **新代码默认选型**         |
| ② `[JsonConstructor]`    | 手动构造 | ❌          | ✅      | ❌        | ✅         | 仅 Json、且类型签名可改    |
| ③ 双构造 + `private set` | 后门     | ❌          | ❌      | ✅        | ✅         | 老项目过渡，**新代码禁用** |
| ④ 自定义 `JsonConverter` | 手写成本 | ❌          | ✅      | ✅        | 手动      | 完全不能动类型签名时救急   |

------------------------------------------------
3. 三句话向团队宣贯
------------------------------------------------
1. **“必填” 从 “人脑记忆” 变成 “编译器语法检查”**——上线前 100 % 拦截，不再靠 Code Review 碰运气。  
2. **“对象不可变” 与 “序列化友好” 不再互斥**——`required init` 一次性满足，再也不用写 700 行构造函数。  
3. **`private set` 不是“只读”，而是“把枪藏抽屉”**——只要项目活到一定规模，就一定会有人拉开抽屉开枪；  
   真只读对象才能永远杜绝线程安全、哈希崩溃、幽灵 bug。

------------------------------------------------
落地 checklist（复制即用）
------------------------------------------------
- [ ] 新建 DTO/VO/Model 统一用 `required` + `init`  
- [ ] 老项目逐步重构：把 `private set` 批量替换成 `init`，删除多余构造  
- [ ] CI 加规则：拦截新增 `private set` 与手写长构造（Sonar / StyleCop / Roslyn Analyzer）  
- [ ] 单元测试覆盖：缺字段反序列化必须抛 `JsonException`，防止回退  

> 结论一句话：  
> **“扔掉构造函数，拥抱 required——让编译器替你守大门，把 bug 杀死在编译器里。”**
